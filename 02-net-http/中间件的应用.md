# 中间件的应用 

## 1. 全局中间件 (Global Middleware)

针对整个路由器”的情况：

- **适用场景**：所有请求都必须经历的操作。比如：请求日志记录（Logger）、全局异常捕获（Recovery）、跨域资源共享（CORS）。

- **Gin 代码示例**：

  ```
  r := gin.New()
  // r.Use() 应用于之后注册的所有路由
  r.Use(gin.Logger()) 
  r.Use(gin.Recovery())
  ```

## 2. 路由组中间件 (Group Middleware)

这是实际开发中最常用的方式之一。我们通常会把某些具有相同属性的接口放在一个组里，然后只给这个组加上中间件。

- **适用场景**：比如 `/api/v1` 需要某种特定的格式化验证，或者 `/admin` 下的所有接口都需要**管理员权限校验**。

- **Gin 代码示例**：

  ```
  // 只有访问 /admin/... 的路由才会经过 AuthMiddleware
  adminGroup := r.Group("/admin")
  adminGroup.Use(AuthMiddleware()) 
  {
      adminGroup.GET("/users", GetUsersHandler)
      adminGroup.POST("/config", UpdateConfigHandler)
  }
  ```

## 3. 单一路由/Handler 专属中间件 (Specific Route Middleware)

**可以单独给某个 handler 设置的** 如果某个前置处理只对某一个特定的接口有意义，我们就会直接把它绑在这个路由上。

- **适用场景**：比如某个特定的高频接口需要单独的限流（Rate Limiting），或者某个特定的下载接口需要做一种特殊的参数解密。

- **Gin 代码示例**：

  ```
  // 在路径和最终的 Handler 之间，可以插入任意多个特定的中间件
  r.GET("/special-data", SpecialDecryptMiddleware(), RateLimitMiddleware(), GetSpecialDataHandler)
  ```

------

### 为什么会有中间件总是在路由器上这种错觉？

因为很多项目在 `main.go` 或者 `router/router.go` 的最顶层，最先写的就是那些全局的基础中间件（比如日志和崩溃恢复），它们确实接管了所有的请求。

但当你深入到具体的业务代码中时，你会发现大量的中间件其实是挂载在具体的 `Group` 或者特定的 `GET/POST` 方法上的。**中间件返回 `Handler` 的设计，正是为了让它可以像拼乐高积木一样，既能拼在最外层，也能精细地拼在某一个特定的零件上。**
