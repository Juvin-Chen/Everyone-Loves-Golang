# Go 架构终极：面向接口的 Repository 模式

## 一、 核心思想：面向“契约”编程，而不是面向“具体实现”

想象一个场景：
* **Handler（业务员）：** 需要存取留言。
* **Repository（仓库）：** 负责具体怎么存（存内存、存 MySQL、存 Redis）。

如果我们不用接口，业务员就等于**直接绑定了某一个具体的仓库管理员（比如老王）**。老王如果辞职了，业务员就傻眼了。
如果使用接口，我们就是**定义了一份《仓库管理员岗位职责》（契约）**。业务员只管照着这份职责说明书下达命令，至于今天来上班的是老王（MySQL）还是小李（本地内存），业务员根本不在乎！

---

## 二、 代码实战：如何用 Interface 改造你的项目？

我们只需分四步，就能完成这次华丽的架构升级。

### 第一步：定义一份“契约”（Interface）
在 `repository` 包里，我们不再直接暴露具体的结构体，而是定义一个接口。只要谁会做这三个动作，谁就是合法的 `MessageRepo`！

```go
package repository

import "message-board-gin/internal/model"

// 💡 契约在此：我不关心你怎么存，我只要你能提供这三个方法！
type MessageRepo interface {
    GetAll() []model.Message
    Add(msg model.Message) error
    DeleteByID(id int) error
}
```

### 第二步：让不同的底层工具去“履行契约”
我们可以写两个完全不同的底层实现（老王和小李）。在 Go 语言里，只要你的结构体实现了接口里的所有方法，你就**自动**成为了这个接口的实例（这叫鸭子类型）。

**实现 A：本地内存版（适合本地测试）**
```go
type MemoryRepo struct {
    data []model.Message
}
func (m *MemoryRepo) GetAll() []model.Message { return m.data }
func (m *MemoryRepo) Add(msg model.Message) error { /* 追加到切片 */ return nil }
func (m *MemoryRepo) DeleteByID(id int) error { /* 从切片删除 */ return nil }

// 构造函数返回接口类型
func NewMemoryRepo() MessageRepo {
    return &MemoryRepo{} 
}
```

**实现 B：MySQL 版（适合正式上线）**
```go
type MySQLRepo struct {
    db *gorm.DB // 假设使用 GORM 连数据库
}
func (m *MySQLRepo) GetAll() []model.Message { /* 执行 SELECT * FROM messages */ }
func (m *MySQLRepo) Add(msg model.Message) error { /* 执行 INSERT */ }
func (m *MySQLRepo) DeleteByID(id int) error { /* 执行 DELETE */ }

// 构造函数同样返回接口类型
func NewMySQLRepo(db *gorm.DB) MessageRepo {
    return &MySQLRepo{db: db}
}
```

### 第三步：Handler 的华丽转身（只认契约）
这是最关键的一步！看仔细，`MessageHandler` 里的 `repo` **不再是具体的指针了，而是接口类型！**（注意：接口类型本身就是引用，不需要加 `*` 号）。

```go
package handler

import "message-board-gin/internal/repository"

type MessageHandler struct {
    // 💡 杀手锏：我只认 MessageRepo 这个接口，不管底层是谁！
    repo repository.MessageRepo 
}

// 注入的时候，传进来的也必须是满足该接口的对象
func NewMessageHandler(repo repository.MessageRepo) *MessageHandler {
    return &MessageHandler{repo: repo}
}

func (h *MessageHandler) Create(c *gin.Context) {
    // 业务逻辑正常写，h.repo.Add() 会自动通过多态调用正确的底层方法
    h.repo.Add(msg) 
}
```

### 第四步：上帝视角的组装（Router 层）
现在，见证奇迹的时刻到了。在 `router.go` （或者你的 `main.go` 组装层）里，你想切换数据库，**只需要改一行代码**！

```go
func Setup() *gin.Engine {
    r := gin.Default()
    
    // 💥 场景 1：今天我们在本地做练习，没装 MySQL，直接用内存版！
    var repo repository.MessageRepo = repository.NewMemoryRepo()
    
    // 💥 场景 2：项目要上线了！只需要注释掉上面那行，换成下面这行！
    // var repo repository.MessageRepo = repository.NewMySQLRepo(dbConnection)
    
    // 注入给 Handler（Handler 内部一行代码都不用改！）
    msgHandler := handler.NewMessageHandler(repo)
    
    r.POST("/messages", msgHandler.Create)
    return r
}
```

## 三、 为什么大厂都必须这么写？

除了上面说的“**无缝切换数据库**”之外，用接口定义 Repo 还有一个在工业界极其重要的作用：**方便写单元测试（Unit Test）**！

假设你想测试 `MessageHandler` 的逻辑对不对，如果你不用接口，你就必须在测试前真正启动一个 MySQL 数据库，这太重、太慢了。
但如果有了接口，你可以在测试代码里随便写一个伪造的 `MockRepo` 塞给 Handler，瞬间就能跑完上万个测试用例！

***

这种把抽象层（接口）和具体实现层剥离开来的思想，其实就是大名鼎鼎的 **SOLID 原则中的“依赖倒置原则 (DIP)”**。
