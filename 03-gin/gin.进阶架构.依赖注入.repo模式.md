# Gin 进阶架构：依赖注入与 Repository 模式

在工业级 / 企业级的 Go Web 开发中，我们绝不会把所有代码都塞在 `main.go` 或者单个函数里。为了让代码**好维护、好扩展、不打架**，我们会引入结构体和分层架构。

## 前言（帮助理解）
为什么要专门弄个 `MessageHandler` 结构体来接收？用大白话解释就是：**为了给这些路由处理函数发一个“装满工具的背包”。**

我们来详细拆解一下，为什么要这么“大费周章”：

#### 1. 痛点：不这么写，你要怎么拿数据库（Repo）？

在 Gin 的路由里，你注册函数只能是这种固定格式：`r.GET("/", 函数名)`。这个函数内部只能接收一个参数：`c *gin.Context`。

现在问题来了：在 `Index` 或者 `Create` 函数内部，你需要去查数据库（也就是代码里的 `repo`）。你怎么拿到这个 `repo`？

- **新手写法（全局变量）：** 在外面定义一个全局的 `var globalRepo *Repository`，然后在函数里直接用。这在大型项目中是**死罪**。全局变量满天飞会导致并发冲突、数据污染，最后根本不知道是哪个文件改了它。
- **笨办法（函数内现造）：** 每次用户请求 `/messages`，都在 `Create` 函数里面重新连接一次数据库，实例化一个 `repo`。这会让你的服务器瞬间卡死，性能极差。

#### 2. 优雅解法：结构体 + 方法接收者 (类似面向对象)

回想一下写 C++ Qt 项目的经验，处理复杂逻辑时，肯定不会把所有槽函数都写成满天飞的全局普通函数。通常的做法是把它们封装成某个窗口类的成员函数（比如 `MainWindow::onButtonClicked()`），然后把需要的数据库指针、配置对象作为这个类的**成员变量**。

Go 语言没有 `class`，它的替代方案就是**结构体 (Struct) + 方法接收者 (Receiver)**。

## 一、 为什么 Handler 要写成结构体？（依赖注入的艺术）

在新手教程中，路由处理函数通常是直接写死的普通函数 `func Create(c *gin.Context)`。但这在真实项目中会遇到致命痛点：**函数内部怎么访问数据库连接？**

* **错误做法：全局变量满天飞。** 定义一个全局的 `DB` 变量。这在并发请求下极其危险，代码容易耦合，且极难写单元测试。
* **错误做法：函数内现连数据库。** 每次请求来都重新建立一次数据库连接，服务器直接原地爆炸。

### 优雅解法：结构体 + 方法接收者



我们通过定义一个结构体，把 Handler 需要的所有“工具”（比如数据库操作对象）当成成员变量装进去。

```go
// 1. 定义一个结构体（相当于给 Handler 发了一个装满工具的背包）
type MessageHandler struct {
    repo *repository.MessageRepository // 数据库操作工具
}

// 2. 这里的 (h *MessageHandler) 就像 C++ 里的 this 指针！
func (h *MessageHandler) Create(c *gin.Context) {
    var msg model.Message
    c.ShouldBind(&msg)
    
    // 3. 完美！直接从 h 里面掏出 repo 工具来操作数据库
    h.repo.Add(msg) 
}
```

**这种模式叫什么？—— 依赖注入 (Dependency Injection)**
我们在外层（`router.go`）把 `repo` 初始化好，然后像塞电池一样“注入”到 `MessageHandler` 中。这样 Handler 就再也不用自己去管怎么连数据库了。

## 二、 为什么要抽离出一个 `Repository` (仓库层)？

在上面的代码中，`h.repo.Add(msg)` 为什么不直接写成 `db.Exec("INSERT INTO ...")`？为什么要专门搞一个 `repository` 包？



这也是工业界的一项铁律：**隔离变化，各司其职。**

把数据库操作单独抽成 Repository 有三大核心优势：

### 1. 职责分离 (解耦)
* **Handler（服务员）：** 只负责和 HTTP 打交道。解析前端发来的 URL、Header、Body，做参数校验，最后把结果打包成 JSON 或 HTML 返回给前端。**它不应该知道任何关于 SQL 的事情。**
* **Repository（仓库管理员）：** 只负责和数据库打交道。执行 `INSERT`、`SELECT`。**它不应该知道任何关于 HTTP、`*gin.Context` 的事情。**
如果把 SQL 直接写在 Handler 里，代码就会又长又臭，极难阅读。

### 2. 代码高度复用
假设你的项目中，除了“留言板”接口需要查所有留言，后台的“管理员面板”也需要查所有留言。
* 如果你把 SQL 查数据库的代码写死在 `Index` Handler 里，那你在写管理员接口时，就得把那一坨 SQL 代码再复制一遍。
* 有了 Repository，你只需要在任何需要的地方调用一句 `h.repo.GetAll()` 即可。

### 3. 无痛替换底层存储
这是最牛的一点！假设你现在的项目初期用的是内存切片（`[]Message`）来存数据，后来项目做大了，需要换成 **MySQL** 数据库。
* 如果你没用 Repository，你需要全文件搜索所有的 Handler，一行行改代码。
* 如果你用了 Repository，**你只需要修改 `repository` 包里的代码**，把里面的逻辑换成 MySQL 语句。外层的 `MessageHandler` 一行代码都不用改！因为它只认 `repo.Add()` 这个动作，根本不在乎你底层是怎么存的。

---

## 三、 总结：标准的 Web 后端架构雏形

经过上面这两个改造，你的项目其实已经具备了现代后端架构的雏形：

1. **路由层 (Router)：** 负责指路，并完成依赖注入（把 Repo 塞给 Handler）。
2. **控制层 (Handler)：** 负责解析 HTTP 请求，呼叫下层干活，返回响应。
3. **数据层 (Repository)：** 纯粹的数据操作，隐藏了具体的 SQL 或存储细节。
