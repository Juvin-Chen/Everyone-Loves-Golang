# Conetxt自定义包装器

## 一、 痛点：原生的 `c.Get` 有两宗罪

在 Go 语言里，`gin.Context` 的内部其实是用一个类似 `map[string]any`（或者叫 `map[string]interface{}`）的结构来存便利贴的。

这就导致了两个极其头疼的问题：

1.  **“魔法字符串”满天飞：**
    保安存的时候写了 `c.Set("userId", 1001)`，大厨取的时候手一抖写成了 `c.Get("userID")`（注意大小写）。编译完全不会报错，但一运行程序就找不到数据，这种 Bug 查起来极其浪费时间。
2.  **丑陋且危险的类型断言：**
    因为 `c.Get` 返回的是 `any` 类型（就相当于 C++ 里的 `void*` 万能指针），大厨每次拿出来都必须强转：`uid := value.(int)`。
    **如果保安不小心存了个字符串 `"1001"`，大厨强转 `int` 时，整个 Go 程序会直接 Panic（崩溃死机）！**

为了消灭这两个隐患，我们就需要写一个“**上下文包装器 (Context Wrapper)**”，给它穿上一层强类型的铠甲。

## 二、 优雅解法：自定义强类型 Context 工具包

我们不要让保安和大厨直接操作 `c.Set` 和 `c.Get`，而是专门新建一个文件（比如叫 `pkg/ctxutil/user.go`），写两个专门的小助手函数来帮他们干活。

### 1. 统一定义暗号（常量）
首先，把便利贴的名字定义成**常量**，彻底消灭拼写错误。
```go
package ctxutil

import "github.com/gin-gonic/gin"

// 私有常量，外部无法篡改，杜绝魔法字符串！
const currentUserIDKey = "app_current_user_id"
```

### 2. 写一个专用的 Setter（给保安 / 中间件用）
```go
// SetUserID 将用户 ID 安全地存入上下文
func SetUserID(c *gin.Context, userID int) {
    // 底层还是用的 c.Set，但对外暴露的是强类型的入参 (int)
    c.Set(currentUserIDKey, userID)
}
```

### 3. 写一个专用的 Getter（给大厨 / Handler 用）
这是最核心的一步！我们把脏活、累活（类型断言、错误判断）全部封装在这个函数里。
```go
// GetUserID 从上下文中安全地提取用户 ID
// 返回两个值：第一个是真实的 ID，第二个是是否提取成功
func GetUserID(c *gin.Context) (int, bool) {
    // 1. 先用常量去拿，绝对不会拼错
    value, exists := c.Get(currentUserIDKey)
    if !exists {
        return 0, false
    }

    // 2. 安全的类型断言！
    // 加上 ok 之后，就算类型不对，程序也不会 Panic，只会让 ok 变成 false
    uid, ok := value.(int)
    if !ok {
        return 0, false
    }

    return uid, true
}
```

## 三、 实战对比：穿上铠甲后的代码有多爽？

现在，我们把这套工具放进你的流水线里，看看代码发生了什么质的飞跃：

### 改造后的中间件（保安）
以前：`c.Set("userID", 1001)`
现在：
```go
func AuthMiddleware() gin.HandlerFunc {
    return func(c *gin.Context) {
        // ... 解析 Token ...
        userID := 1001 
        
        // 极其清晰，而且只允许传 int，传错类型编译器直接报错！
        ctxutil.SetUserID(c, userID) 
        
        c.Next()
    }
}
```

### 改造后的 Handler（大厨）
以前：四五行又臭又长的判断和断言，满屏的 `interface{}`。
现在：
```go
func (h *MessageHandler) Create(c *gin.Context) {
    // 爽！一行代码安全拿到真实的 int 类型 ID，没有魔法字符串，没有类型断言！
    uid, ok := ctxutil.GetUserID(c)
    if !ok {
        c.JSON(401, gin.H{"error": "获取用户身份失败"})
        return
    }
    
    // 直接拿着 uid 去调用你的 Repo
    h.repo.AddMessage(uid, "留言内容")
    c.JSON(200, gin.H{"msg": "成功"})
}
```

## 核心总结

这就是工业界常说的“**防备型编程**”。

原生的 `c.Get` 就像一个盲盒（`any`），谁都可以往里面塞任何东西。通过我们封装的 `ctxutil.GetUserID`，这个盲盒变成了一个**“只能装硬币的专属投币口”**，不仅安全，而且让真正写业务逻辑的 Handler 变得极其干净利落。

这套从“路由组装 -> 闭包传参 -> 中间件设值 -> 包装器取值 -> Handler 干活 -> Repo 存库”的完整链路，逻辑已经闭环
