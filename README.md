# go backend 

在 Go 语言里，只要需要引入外部的第三方库（比如 Viper、Gin、GORM 等），或者想把代码按不同文件组织起来，**必须**先初始化一个 **Go Module**（模块）。否则 Go 找不到这些库，终端就会一直报错。

### 第一步：初始化模块 (go mod init)

在你的 **项目** 文件夹下打开终端，输入以下命令来初始化这个项目：

```
go mod init xxx-demo
```

*(注：`xxx-demo` 是这个模块的名字，可以随便起)*

### 第二步：下载第三方库 (go get)

有了 `go.mod` 撑腰，就可以放心大胆地下载第三方库了。在终端继续输入：

`for example:`

```
go get github.com/spf13/viper
```

### 第三步：观察变化

下载完成后，会发现两件事：

1. `go.mod` 文件里面多了一行代码，记录了你下载的 viper 版本。
2. 文件夹里多了一个 `go.sum` 文件。这个不用管它，它是 Go 用来校验下载的库有没有被篡改的安全文件。

### 解惑：电脑里有了库，为什么每次还要 `go get`？

简单来说：**库只会在电脑里真正下载一次，但每个项目都必须显式地“声明”自己要用它。**

1. **全局缓存 (帮你省时间)：** 当你第一次在某个项目里 `go get gorm.io/gorm` 时，Go 会从网上把它下载下来，存到你电脑的一个全局缓存大仓库里（通常在你的 `GOPATH/pkg/mod` 目录下）。
2. **项目隔离 (保证不出错)：** 当你新建一个文件夹（比如 `05-gorm-test`）并初始化 `go.mod` 后，再次执行 `go get`，Go **不会**去网上重新下载，而是直接去你的全局缓存里找。
3. **为什么必须声明？** 因为 Go 强调**工程化**。假设你半年前写的老项目用的是 GORM v1.20，现在的新项目想用 GORM v1.25。如果 Go 不强制每个项目独立管理自己的 `go.mod`（依赖清单），不同版本的库就会在电脑里打架。所以，每次开新文件夹，执行 `go get` 其实是在往这个特定项目的 `go.mod` 清单里“登记”版本号，而不是傻傻地重新下载。
