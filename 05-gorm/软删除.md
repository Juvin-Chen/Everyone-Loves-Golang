# 结合共享雨伞项目理解｜GORM 软删除 

## 核心结论
当调用 `db.Delete()` 删除雨伞，**数据却还在数据库里**，不是代码错了，是 GORM 开启了**软删除**！
软删除 = **假删除**，数据保留，只打「已报废」标记，普通查询看不见，历史数据永久留存。

---

## 1. 揭秘：为什么删了数据还在？
罪魁祸首就是你结构体里的 **`gorm.Model`**！
```go
type Umbrella struct {
	gorm.Model  // 自带4个字段：ID、CreatedAt、UpdatedAt、DeletedAt
	SerialNumber string // 雨伞编号
	Status       string // 状态
}
```
`gorm.Model` 里藏了一个关键字段：**`DeletedAt`（删除时间）**
- 只要有这个字段，GORM 就会默认开启软删除
- 你执行 `db.Delete()`，GORM **不会真删数据**，只会执行：
  ```sql
  UPDATE umbrellas SET deleted_at = '当前时间' WHERE id = 伞ID;
  ```
- 数据完好无损地留在表里，只是被标记为「已删除」

---

## 2. 软删除 VS 物理删除（硬删除）
| 类型 | 操作行为 | 数据状态 | 能否找回 |
| ---- | -------- | -------- | -------- |
| **软删除（GORM默认）** | 标记删除时间 | 数据还在，只是隐藏 | 随时找回 |
| **物理删除（真删除）** | 直接删除数据 | 数据彻底消失 | 永久丢失 |

简单说：
**软删除 = 把雨伞扔进储藏室，看不见但还在**
**物理删除 = 把雨伞直接扔掉，再也找不到**

---

## 3. 共享雨伞项目，为什么必须用软删除？
这是企业级项目的**刚需**：
1. **保证关联数据不报错**
   一把伞报废了，之前的**借阅记录**还关联着这把伞的ID
   如果真删了，借阅记录会变成死数据，直接报错
2. **方便财务/运营对账**
   月底统计：一共采购多少伞？报废多少伞？
   软删除能保留所有历史数据，死有对证
3. **防止误操作**
   手滑删错数据，软删除能轻松找回，物理删除直接凉凉

---

## 4. 代码实操（一看就会）
### ① 软删除（标记报废，数据保留）
```go
// 报废ID=2的雨伞 → 软删除，数据还在
db.Delete(&Umbrella{}, 2)
```
效果：`deleted_at` 填充时间，普通查询查不到这把伞

### ② 普通查询（只查「未报废」的伞）
```go
// GORM 自动过滤 deleted_at 有值的数据
var umbrellas []Umbrella
db.Find(&umbrellas)
```

### ③ 查询所有数据（包含已报废的伞）
```go
// Unscoped()：取消软删除过滤
var allUmbrellas []Umbrella
db.Unscoped().Find(&allUmbrellas)
```

### ④ 物理删除（彻底抹除，找不回）
```go
// 测试数据写错了，才用这个！真·删除
db.Unscoped().Delete(&Umbrella{}, 2)
```

---

## 5. 一句话终极总结
1. **`gorm.Model` 自带 `DeletedAt` → 自动开启软删除**
2. **`db.Delete()` 不是删数据，是标记「已删除」**
3. **共享雨伞项目必须用软删除，保数据、防报错、好对账**
4. **想真删数据，必须加 `Unscoped()`**
